**Date:** 2026-05-03
**Author:** Claude Code (with Varun)

**Versioning:** target versions are decided at release time, not in this plan. Each milestone is a shippable chunk; we may ship them as separate releases, combine them, or reorder based on what's actually done when we cut a version.

**Milestones (in current proposed order):**

2. **M2 — content sources** (1.5 days): widget area inside mega, auto-populated post columns

## Audit: actual header styles vs plan assumptions

Audited `resources/views/sections/header.blade.php` (838 lines, 8 style branches) against the plan. Five mismatches between what the plan assumed and what the codebase actually does. Each is addressed in the milestone scope below.

### Inventory: 8 header styles + their mobile patterns

| # | Style | Desktop nav | Mobile pattern | Mega-anchor target |
|---|---|---|---|---|
| 1 | `minimal` | NONE — only floating hamburger top-right (always visible) | Fullscreen `#brndle-minimal-overlay` (vertical column, opacity fade) | n/a — no horizontal nav exists |
| 2 | `centered` | Logo top center, nav as **rounded pills** with gap-1, below the logo | `#brndle-mobile-menu` collapse below nav | each pill `<li>` |
| 3 | `transparent` | Fixed, transparent until scroll — logo left, nav inline, CTA right | `#brndle-mobile-menu` collapse | nav row (single row) |
| 4 | `solid` | Relative, solid bg, border-bottom — logo left, nav inline, CTA right | `#brndle-mobile-menu` collapse | nav row |
| 5 | `split` | **TWO ROWS** — top: logo + search + CTA. Bottom: nav inline left, dark-mode toggle right | `#brndle-mobile-menu` collapse | **bottom nav row only**, not the top row |
| 6 | `banner` | Promo banner above + standard nav row below | `#brndle-mobile-menu` collapse | nav row (banner adds vertical offset, mega's `position: absolute` from the nav strip handles it naturally) |
| 7 | `glass` | Fixed, frosted backdrop-blur, gradient top border | `#brndle-mobile-menu` collapse | nav row — but mega panel needs **its own opaque bg** so the parent's blur doesn't leak through |
| 8 | `sticky` (default `@else`) | Standard horizontal sticky on scroll | `#brndle-mobile-menu` collapse | nav row |

### Mismatch 1 — Plan assumed drawer-style mobile menu. Reality: 7/8 use collapse-down accordion, 1/8 (minimal) uses fullscreen overlay. No off-canvas drawer exists today.

**Plan's old wording** said "Mobile drawer animation — `slide-right` / `slide-left` / `full-screen` / `accordion-in-place`" as if drawer was the default and accordion was an option. **Reality:** accordion-in-place IS the only pattern in 7 styles, and minimal already has its own fullscreen overlay (which is essentially a vertical-axis variant of `full-screen`).

**Resolution:**
- M1 honors the existing patterns. The walker emits nested-item disclosure markup INSIDE the existing `#brndle-mobile-menu` collapse panel — it does not replace the collapse itself.
- Drawer animation choices (slide-from-side) move to a separate, later proposal IF requested. Not in M1–M4.
- The `minimal` overlay is treated as a flat-list context — mega config is ignored, all items render with their description + icon stacked vertically.

### Mismatch 2 — Plan didn't account for `minimal` having no horizontal nav.

The mega-menu concept (a panel that drops below a top-level item) doesn't visually apply when there IS no top-level item visible — minimal hides the entire nav until the overlay opens.

**Resolution:**
- The walker detects `headerStyle === 'minimal'` (passed in via constructor) and unconditionally flattens mega config to standard list rendering.
- Admin UI shows a hint above the mega fields: "Mega menu config is rendered as a flat list when the active header style is Minimal (no horizontal nav strip)." Doesn't disable the fields — just sets expectations.

### Mismatch 3 — Plan didn't account for `split` having 2 rows.

In `split` style, the menu lives in the BOTTOM row (line 460ish in header.blade.php), but the visible header height includes the top row too. Mega panel anchored to `position: absolute; top: 100%` of the nav UL will drop below the bottom row correctly — but if anyone tried to position it relative to the `<header>` element it would overlay the search box.

**Resolution:**
- All mega CSS uses `position: relative` on the nav `<ul>` and `position: absolute` on the mega panel relative to that UL. Header-level positioning is never used.
- Documented as a constraint in the CSS comments so future header styles don't break it.

### Mismatch 4 — Plan glossed over backdrop-blur conflicts in `glass` and `transparent`.

`glass` uses `bg-surface-primary/80 backdrop-blur-2xl backdrop-saturate-150` on the header bar. `transparent` uses `bg-transparent` until scroll. If the mega panel inherited or sat behind those treatments, the blur would leak into the panel content.

**Resolution:**
- `.brndle-mega` always sets its own `background: var(--color-surface-primary)` (opaque, not the `/80` translucent variant).
- Adds a small drop shadow `box-shadow: 0 8px 24px rgba(0,0,0,0.08)` so the panel reads as a discrete surface above the blurred header.
- For `transparent` style specifically, when the page hasn't scrolled yet, the header is invisible but the mega panel still has its own opaque bg — looks deliberate, not accidental.

### Mismatch 5 — Plan didn't acknowledge that mobile menus today have ZERO nested-item support.

Today the mobile menus just stack flat links, even when the WP menu has parent/child structure. There's no disclosure button, no expand/collapse, no aria-expanded. So adding mobile accordion isn't a "fallback" — it's genuinely new behavior for all 8 header styles.

**Resolution:**
- M1's walker, when in mobile context (constructor flag), emits a `<button aria-expanded="false">` toggle next to any parent `<a>`, plus the children inside a `[hidden]` `<ul>` that the JS un-hides + animates `max-height`.
- This is new code, not a tweak. Time estimate for M1's mobile work bumped from 0 to ~4 hours.
- Existing single-script-per-style toggle handlers stay untouched. The new module attaches to `[data-brndle-disclosure]` elements only and doesn't conflict.

### Mismatch 6 (minor) — Existing inline scripts per style

There are 7 separate inline `<script>` blocks in header.blade.php (one per style except minimal which has its own scheme). Each handles the mobile menu open/close. The mega-menu module needs to NOT fight those.

**Resolution:**
- Mega-menu JS is event-scoped to elements with `[data-brndle-has-submenu]` and `[data-brndle-disclosure]`. It never touches `#brndle-menu-btn` or `#brndle-mobile-menu` directly.
- Existing scripts continue to work. This is consolidation territory — defer to a later cleanup release if the user wants one module instead of 7.

### Z-index audit

| Element | Current z | Notes |
|---|---|---|
| `#brndle-banner` (banner style) | `z-50` | promo strip |
| `#brndle-header` (all others) | `z-50` | header bar |

## Why

Brndle ships 8 header styles but every `wp_nav_menu()` call in `resources/views/sections/header.blade.php` uses the default walker with no submenu CSS. Result: menus that have child items render the children inline as a raw nested `<ul>`, which means **2-level nesting is broken today, 3-level doesn't exist, and there's no mega-menu story at all**. For a theme positioned as enterprise-ready (the BuddyX / Reign / B2B SaaS audience), this is a blocking gap. Real product sites need mega menus — Stripe, Shopify, Linear, Vercel all use them.

This plan covers the full spectrum so we ship one cohesive system rather than three half-features:
- Standard 2-level dropdown (default for any submenu without mega config)
- 3-level fly-out (for content sites with deeper IA)
- Mega menu (multi-column panel with featured block + CTA — opt-in per top-level item)

## What we are NOT doing

- We are not replacing the WP nav menu admin UI. Brndle hooks into the existing menu editor with custom fields per item. No separate "Mega Menu Builder" screen.
- We are not building a posts/products query block inside the mega panel (defer to 1.6.x once the foundation ships).
- We are not adding shortcode support inside mega panels (security + complexity; defer).
- We are not adding animation libraries — vanilla JS, CSS-only transitions, ~3KB total.

## Architecture

### Storage: per-menu-item meta

Every WP nav menu item is a post (`nav_menu_item` post type), so per-item config goes in post meta. Keys:

```
_brndle_mega_menu          bool    only on depth=0 items
_brndle_mega_columns       int     2-4
_brndle_mega_featured_image int    media ID
_brndle_mega_featured_heading       string
_brndle_mega_featured_description   string
_brndle_mega_featured_url   string
_brndle_mega_cta_text       string
_brndle_mega_cta_url        string
_brndle_column              int     1-4 (per child)
_brndle_column_heading      string  if non-empty, item renders as <h4> heading not link
_brndle_icon                string  Lucide icon name (kebab-case)
_brndle_description         string  1-line description
_brndle_badge               string  NEW / BETA / PRO / custom
```

Hook: `wp_nav_menu_item_custom_fields` (WP 5.4+) for the admin form, `wp_update_nav_menu_item` for save. Sanitizer in `MenuItemMeta::save()` clamps integers, validates enum (badge), runs `sanitize_text_field()` / `esc_url_raw()`.

### Walker: `Brndle_Nav_Menu_Walker`

Extends `Walker_Nav_Menu`. Methods:

- `start_lvl()` — opens nested wrapper. If parent has `_brndle_mega_menu=1` and depth=0, emit `<div class="brndle-mega" data-cols="N">` and a `<div class="brndle-mega__columns">`. Otherwise emit `<ul class="sub-menu">`.
- `end_lvl()` — closes wrapper. For mega: also emit featured block + CTA row before closing.
- `start_el()` — emits the `<li>` + link. Inside: icon span, title, badge, description. Detects column heading mode and emits `<li class="brndle-mega__heading">` with `<h4>` instead of link.
- `end_el()` — closes `</li>`.
- Stateful: tracks the parent's mega config in a per-instance property so children know what context they're in.

The walker auto-detects context (mega vs standard) and produces correct markup either way. Same call site, no new arguments needed in `header.blade.php` — just add `'walker' => new Brndle_Nav_Menu_Walker()`.

### Frontend JS: `resources/js/mega-menu.js`

Vanilla, ~2.5KB minified. One module that:

1. Discovers every top-level menu item with children: `document.querySelectorAll('[data-brndle-has-submenu]')`
2. Adds ARIA: `aria-haspopup="true"`, `aria-expanded="false"`, `role="menuitem"` on triggers; `role="menu"` on submenus
3. Hover open (100ms debounce, prevents flicker on fast cursor moves)
4. Click open (touch/keyboard parity)
5. Click-outside / Escape closes
6. Tab through items in document order
7. Arrow keys: ← / → between top-level, ↓ to enter submenu, ↑ to leave, Esc to close
8. Mobile: detects `(hover: none)` — tap toggles instead of hover, mega becomes accordion

No framework. Idempotent attach (guarded by `window.brndleMegaMenuInit`).

### CSS: `resources/css/app.css` (additions)

New scoped block (~200 lines). Three breakpoints + a hover-capability check, mapped to real device classes rather than guessed widths:

| Range | Device class | Behavior |
|---|---|---|
| `< 640px` | phone (iPhone SE / 14 / 14 Pro Max, Android small/large) | Drawer accordion only. Tap toggles. Tap targets ≥ 44×44. Close button bottom-aligned for thumb reach. |
| `640px – 1023px` | tablet portrait + small landscape (iPad mini 744, iPad 820, iPad Pro 12.9 portrait 1024) | Drawer accordion. Drawer is wider (max 480px) but layout is identical to phone. |
| `≥ 1024px` | tablet landscape + desktop (iPad landscape 1180, iPad Pro 12.9 landscape 1366, laptops, desktops) | Full mega panel. Hover triggers IF the device supports hover. |
| `(hover: none)` (any width) | touch-primary regardless of viewport size | No hover triggers anywhere. Tap toggles open + ARIA `aria-expanded` flips. iPad-with-finger hits this; iPad-with-trackpad does not. |

```css
/* Standard 2-level dropdown — desktop only */
@media (min-width: 1024px) {
  .brndle-nav .sub-menu { position: absolute; ... }
  .brndle-nav .sub-menu .sub-menu { left: 100%; top: 0; ... } /* fly-out right */
}

/* Mega panel — desktop only */
@media (min-width: 1024px) {
  .brndle-mega { position: absolute; left: 0; right: 0; ... }
  .brndle-mega__columns { display: grid; grid-template-columns: ... }
}

/* Tablet portrait + small landscape: drawer behavior identical to phone,
   but wider drawer panel for better tap-target spacing */
@media (max-width: 1023px) and (min-width: 640px) {
  .brndle-mobile-drawer { max-width: 480px; }
}

/* Phone: drawer fills viewport; iOS Safari needs dvh for the bottom-toolbar
   resize jump. min-height as fallback for browsers without dvh support. */
@media (max-width: 639px) {
  .brndle-mobile-drawer {
    min-height: 100vh;
    min-height: 100dvh;
    overscroll-behavior: contain; /* prevent rubber-band into page underneath */
  }
}

/* Tap targets: every menu link / disclosure button hits 44×44 minimum on
   touch. Apple HIG + WCAG 2.5.5 + Material 48dp guidance converge here. */
.brndle-nav button,
.brndle-mobile-drawer a,
.brndle-mobile-drawer button {
  min-height: 44px;
  touch-action: manipulation; /* eliminates 300ms tap delay on legacy touch */
}

/* Hover-capability check: kill all hover-only triggers when the primary
   pointer can't hover (iPad with finger, phones, kiosk touchscreens). */
@media (hover: none) {
  .brndle-nav [data-brndle-has-submenu]:hover > .sub-menu,
  .brndle-nav [data-brndle-has-submenu]:hover > .brndle-mega { display: none; }
  .brndle-nav [data-brndle-has-submenu][aria-expanded="true"] > .sub-menu,
  .brndle-nav [data-brndle-has-submenu][aria-expanded="true"] > .brndle-mega { display: block; }
}

/* Reduced motion */
@media (prefers-reduced-motion: reduce) {
  .brndle-nav .sub-menu,
  .brndle-mega,
  .brndle-mobile-drawer__panel { transition: none; animation: none; }
}
```

### Mobile drawer

Each header style's mobile drawer already calls `wp_nav_menu` separately (e.g. `'menu_class' => 'space-y-1'`). Pass the same walker — it detects the mobile context via a constructor flag and emits accordion-friendly markup: each parent gets a `<button aria-expanded>` toggle, children render in a `<div hidden>` panel that the JS toggles open via `max-height` transition.

## Risk notes

- **Menu item meta caching:** WordPress's `wp_get_nav_menu_items()` primes post meta cache, so reading `_brndle_*` keys is one query for the whole menu, not per item. Verified for the typical 20-item primary menu.
- **Lucide icon list:** the existing `bin/copy-lucide-icons.mjs` controls which icons ship. The admin icon dropdown reads the same source so users can only pick installed icons. If they want a new one they add it to that list and rebuild — same flow as elsewhere in the theme.
- **8 header styles × 15 wp_nav_menu calls:** every call needs the `'walker' => ...` arg. Easy to miss one. Mitigation: a `bin/check-nav-walker.mjs` script that greps the header file for `wp_nav_menu(` calls and asserts each has `'walker'`.
- **Hover state on touch devices:** desktop-style hover-open menus are a known a11y problem on hybrid devices. The JS detects `(hover: none)` via `matchMedia` and disables hover triggers; tap is the only affordance.
- **Z-index conflicts:** the mega panel needs a z-index higher than dark-mode toggle / search dropdown / View Transitions overlays. We'll use `--z-mega: 90;` (below modals at 100, above page content at 10).

## Open decisions (resolve during phase 1)

1. Should "render as section heading" items be focusable via keyboard? Convention: no — they're decorative dividers. Skip with `tabindex=-1`.
2. Should mega panels close when navigating to a child link? Yes — improves perceived speed; close immediately, let view-transition handle the page swap.
3. Should we provide a per-block-style mega variant (e.g. "compact" vs "spacious")? Phase 1: spacious only. Add `data-density` attribute later if requested.
4. Should empty mega columns auto-collapse? Yes — count non-empty columns before emitting and adjust grid template. Avoids visual gaps when admin has fewer items than declared columns.

## Backwards compatibility

- Existing sites: zero behavior change. Walker's standard-dropdown branch produces visually identical markup to the default walker for items without any `_brndle_*` meta. The only addition is `aria-haspopup` and `aria-expanded` on items with submenus — these are additive and don't affect rendering.
- Existing mobile drawers: same. The walker's mobile-context flag preserves the current `space-y-1` markup unless an item has nested children, in which case it adds the disclosure button + collapsible region.
