# Plan: v2.1 — editorial blocks (premium UX)

**Date:** 2026-05-04
**Author:** Claude Code (with Varun)
**Effort:** ~12–14h across 4 milestones (4 new blocks + shared infra)

This plan covers v2.1 of the post-1.9.2 audit roadmap. Four new editorial blocks — code, pull quote, timeline, tabs/accordion — built to a premium UX bar that matches Brndle's existing 14-block design system and follows the wp-block-development quality skill.

## Goals

1. Fill the four obvious gaps in the editorial block library so editors don't reach for plugins.
2. Match (not lower) the polish of the existing 14 blocks. No "good enough" placeholders.
3. Keep the backend Blade-rendered, server-side, apiVersion 3 — Brndle's existing pattern.
4. Make every interactive element WAI-ARIA compliant + keyboard-navigable + reduced-motion safe.
5. Zero new heavy runtime dependencies. Every JS controller stays under 4KB gzipped, lazy-enqueued.

## Constraints (non-negotiable)

From `wp-content/themes/brndle/CLAUDE.md`:
- Tailwind v4 utilities + Brndle theme tokens (`bg-surface-primary`, `text-text-primary`, `text-accent`, `bg-accent`, etc.) — never `text-[var(--color-*)]`.
- Editor JSX → `@wordpress/icons` (auto tree-shaken).
- Frontend Blade → `<x-icon name="kebab-case" />` (Lucide via `bin/copy-lucide-icons.mjs`).
- Block Blade in `resources/views/blocks/` (NOT `blocks/{name}/`).
- `apiVersion: 3` + `save: () => null` (server-side render).
- No emoji as UI affordance. Always use Lucide.
- Editor strings via `__('…', 'brndle')` + `wp_set_script_translations()` already wired.

From `superpowers:wp-block-development`:
- Per-instance unique-id scoping for any block whose CSS isn't fully composable from utilities (timeline connector line, code-block scrollbars, accordion height transition).
- ARIA + keyboard nav + focus indicators + `prefers-reduced-motion` on every interactive surface.
- BEM-ish class naming for the parts that need scoping: `.brndle-{block}__{element}--{modifier}`.
- Per-side spacing where it makes sense (code block padding, timeline rail offset).
- Hover + focus states everywhere a click is possible.
- Theme isolation — every CSS rule selector scoped to the block class.
- No jQuery — vanilla only.

From memory:
- **No SEO schema competition** — no Article / BreadcrumbList / Person JSON-LD. The existing FAQ block emits `FAQPage` schema and that's the only theme-emitted graph. The new tabs/accordion block must NOT emit `FAQPage` — that's what FAQ is for.
- **No versions in plans** — release tag gets assigned at release time.
- **Long-term solutions over per-instance patches** — build a shared block CSS layer + Blade component for repeated patterns (icon button, scoped wrapper) rather than copy-paste per block.

## Aesthetic direction

Editorial register, not landing-page register. These blocks live inside long-form articles, not above the fold:

- Generous vertical rhythm (1.6 line-height in body, 1.2 in headings).
- Restraint on color — accent only for affordances (copy button, active tab indicator, tag chips), never for decoration.
- Hairline dividers (`border-border-subtle`), not heavy borders.
- Typography-first hierarchy — let typography carry the design, not boxes / shadows.
- Reduced-motion paths neutralize transitions completely (no "subtle" 80ms fades — full opacity-only swaps).

Reference designs (each block picks two from this set as North Star):

- **Code block**: GitHub blob view (light), Stripe Docs (light + dark), Vercel changelog inline code, Linear changelog blocks.
- **Pull quote**: New York Times opinion column, The Verge feature articles, Stripe Press long-form.
- **Timeline**: Linear changelog, Stripe Atlas roadmap, Notion changelog, Vercel changelog.
- **Tabs/Accordion**: Radix UI primitives docs, Stripe Docs, GitHub Docs, Linear changelog FAQ.

## Cross-cutting infrastructure

Built once, used by all four blocks:

### Shared Blade component: `<x-block-wrapper>`
`resources/views/components/block-wrapper.blade.php`. Standardizes:
- Outer `<div>` with `class="brndle-{block} {{ $align ?? '' }}"`.
- Per-instance `id="brndle-{block}-{uniqueId}"` from `$attributes['uniqueId']` (auto-generated client-side via `wp.data.select('core/block-editor').getBlock(...).clientId.slice(0, 8)`).
- Optional anchor: passes through `$anchor` from `supports.anchor`.
- Keeps every new block's outer markup identical so `@media print` rules and `view-transitions` selectors stay simple.

### Shared client-side `uniqueId` attribute pattern
Every new block declares `attributes.uniqueId` (string, default `''`). On first edit a `useEffect` sets it to `clientId.slice(0, 8)` if empty. Keeps SSR markup deterministic across re-renders + lets per-instance CSS scope safely.

### Shared editor helpers: `blocks/src/components/`
- `inspector-spacing.js` — exports `<SpacingControls>` (top/right/bottom/left ranges, per-breakpoint, optional). Blocks that use spacing pass + persist a `spacing: {desktop, tablet, mobile, paddingY, paddingX}` attribute. Renderer translates to Tailwind utilities or inline style fallback.
- `inspector-align.js` — exports `<AlignControls>` for blocks with multiple alignment variants (pull quote).
- `clipboard-button.js` — small JSX `<CopyButton text>` used by code block; renders a Lucide `copy` icon button with a `Copied!` toast that fades after 1.6s. Already follows reduced-motion.

Both are JSX-only (used in editor); the matching frontend behavior is plain vanilla in `resources/js/blocks/`.

### Lucide icon set additions
`bin/copy-lucide-icons.mjs` adds:
- `copy` — code block copy affordance
- `code` — code block editor icon (also used as inspector icon)
- `quote` — pull quote
- `circle` — timeline node default
- `chevron-up` — accordion collapse state (have `chevron-down` already)

`npm run icons:copy` after edit, then commit.

### Shared CSS scope: `resources/css/blocks/`
New directory, four small CSS files (one per block) `@import`ed from `resources/css/app.css`. Every rule scoped to `.brndle-{block}`. Pulls Brndle tokens via `var(--color-*)`. No bare element selectors.

Why not utilities only? — code-block scrollbar, timeline connector line, accordion height transition, copy-button toast all need real CSS. Utilities don't reach there.

## Per-block specs

### M1 — Code block (`brndle/code`)

**Reference:** GitHub blob view + Stripe Docs.

**What:** Inline-readable, fixed-width, syntax-highlighted code container with copy button. Replaces core's `core/code` for technical posts. Coexists with core/code (editors who want plain-pre still have it).

**Attributes (`block.json`):**
- `code` (string) — raw code text
- `language` (string, default `'plain'`) — one of: `plain`, `js`, `ts`, `jsx`, `tsx`, `php`, `python`, `bash`, `shell`, `css`, `scss`, `html`, `json`, `yaml`, `markdown`, `sql`, `diff`, `nginx`, `dockerfile`. (18 + plain.)
- `showLineNumbers` (boolean, default `false`)
- `showCopy` (boolean, default `true`)
- `caption` (string, default `''`) — optional one-line caption rendered below the code (cite source / file path / commit SHA).
- `theme` (enum: `auto`, `light`, `dark`, default `auto`) — explicit theme override; `auto` follows `[data-theme]`.
- `uniqueId` (string)

**Editor UX:**
- Placeholder: dashed-border CodeEditor with "Paste or type your code…" placeholder, language dropdown to the right of the toolbar (BlockControls).
- BlockControls: language dropdown (uses `<ToolbarDropdownMenu>` with the 18 languages + `plain`), copy-toggle button, line-numbers toggle.
- InspectorControls: `<TextareaControl>` for caption, `<RadioControl>` for theme override, `<ToggleControl>` for line numbers + copy button.
- Editor preview uses `ServerSideRender` (consistent with other Brndle blocks). Highlightjs is **not loaded in the editor canvas** — preview shows plain monospaced text on a tinted background. Simpler + faster.

**Frontend UX:**
- Renders `<pre><code class="hljs language-{lang}">…</code></pre>` server-side; highlighter runs client-side after first paint.
- highlight.js loaded **lazily** as an ES module from CDN (`esm.sh/@highlightjs/cdn-assets@11.10.0`) via dynamic `import()` ONLY when a `.brndle-code` element is in the DOM (intersection observer triggers within 200px of viewport — never load if user never scrolls to it).
- Filter `brndle/code_highlighter_strategy` — `'cdn'` (default) | `'inline'` (use bundled subset of languages from `node_modules/highlight.js`) | `'php'` (server-side via `Highlight.php` if installed). Documented in the changelog as power-user-tier.
- Copy button: bottom-right of the code container, fades in on `:hover` of the wrapper, always-visible on `:focus-within`. Click → `navigator.clipboard.writeText()` → button label flips to `Copied!` for 1.6s with Lucide `check`. Falls back to a `document.execCommand('copy')` path on older browsers but doesn't bother showing a fallback message (toast is the only UX).
- Line numbers (when on): rendered as a separate column with `aria-hidden="true"` so screen readers don't read them. Selectable but excluded from clipboard via `user-select: none` on the line-number column.
- Long lines: horizontal scroll with `tab-size: 2`. Mobile: scroll snap to start.
- Caption: `<figcaption>` below the `<pre>`, italic, text-tertiary, 0.875rem.
- The `<pre>` is `tabindex="0"` with `role="region" aria-label="Code block: {language}"` so keyboard users can scroll horizontally.
- Reduced motion: copy-button fade is `0ms`, toast is `transition: none`, replaced by an instant text swap.

**Edge cases:**
- Empty `code`: don't render the block at all (returns empty string).
- `language` not in the curated list: render with `class="hljs"` only — highlighter ignores, code stays unstyled but readable.
- highlightjs CDN fetch fails: code stays as plain monospaced text with token coloring missing. No console error; one-time `console.warn` for diagnostics.
- print: code block keeps monospace + pre-wrap; no copy button (already in print stylesheet).

**Tests / verification:**
- Insert in editor → switch language → reload → markup persists.
- View on frontend → highlightjs loads → copy button works → `Copied!` toast appears.
- Disable JS → code still readable, copy button hidden, line numbers visible.
- Tab into code block → arrow keys scroll horizontally.
- print preview → no copy button, monospace preserved.

### M2 — Pull quote (`brndle/pull-quote`)

**Reference:** New York Times opinion column + The Verge.

**What:** Editorial pull quote. Three visual variants. Cite optional. Replaces editors reaching for `core/pullquote` or `core/quote` styled inline.

**Attributes:**
- `quote` (string) — the quoted text
- `cite` (string, default `''`) — attribution
- `citeUrl` (string, default `''`) — optional hyperlink on the cite
- `variant` (enum: `bordered-left`, `centered-large`, `outset`, default `bordered-left`)
- `align` (enum: `none`, `left`, `right`, `wide`, `full`, default `none`) — via `supports.align`
- `accentColor` (enum: `accent`, `text-primary`, `text-tertiary`, default `accent`) — used by `bordered-left` for the rule + the open-quote glyph
- `uniqueId` (string)

**Variants:**
- `bordered-left`: 4px accent rule on left, quote at 1.5rem italic, cite below in small caps.
- `centered-large`: full-width centered, 2.5rem display weight, decorative open-quote glyph above (Lucide `quote` at 32px in the accent color), cite center-aligned.
- `outset`: quote text breaks out of the article column by ~80px on each side at `lg+`. Mobile collapses to centered-large. Useful for long-form features.

**Variants registered as `block.json` `variations`** so the inserter shows three preview tiles.

**Editor UX:**
- RichText editing for `quote` + `cite` (inline formatting allowed: bold, italic, link).
- BlockControls: variant cycle button (cycles through 3), alignment toolbar (uses `supports.align`).
- InspectorControls: `<TextControl>` for cite URL, `<RadioControl>` for accent color (3 swatches).

**Frontend UX:**
- Semantic `<figure>` > `<blockquote>` > `<p>` + `<figcaption>` (when cite present). Cite URL becomes an `<a rel="nofollow"` if present.
- `bordered-left` border uses `border-l-4 border-accent` — but if variant accent is `text-primary`, use `border-text-primary`. Build a small CSS class map.
- `centered-large` quote glyph: Lucide `quote` SVG with `aria-hidden="true"`.
- `outset` uses `margin-inline: -80px` at `lg+` — clamps to `margin-inline: 0` below `lg`. Tested against the 700px article column the standard layout uses (so the outset extends to ~860px width).
- Reduced motion: no animations needed (block is purely static).

**Edge cases:**
- Empty `quote`: don't render.
- No `cite`: omit the `<figcaption>` entirely (don't render an empty container).
- `outset` inside narrow layouts (`sidebar` single-post layout that has 600px column): falls back to `centered-large` via a CSS `:where(.sidebar) .brndle-pull-quote.is-outset` override.

**Tests:**
- All 3 variants render correctly at desktop + tablet + mobile.
- Cite link clickable, opens correctly.
- `outset` variant doesn't break the article column at 700px width (no horizontal scroll on the page).

### M3 — Timeline (`brndle/timeline`)

**Reference:** Linear changelog + Stripe Atlas.

**What:** Vertical list of milestones, each with date + title + description + optional icon. Connector line between dots.

**Attributes:**
- `title` (string, default `''`) — optional section title
- `items` (array of `{ date: string, title: string, description: string, icon?: string }`) — each item
- `iconStyle` (enum: `dot`, `numbered`, `lucide`, default `dot`) — global icon treatment
- `connector` (enum: `solid`, `dashed`, `none`, default `solid`)
- `density` (enum: `compact`, `comfortable`, default `comfortable`)
- `uniqueId` (string)

**Editor UX:**
- Reuses the FAQ block's repeater pattern (PanelBody per item + add button).
- Per-item icon picker (when `iconStyle === 'lucide'`): `<TextControl>` with `<datalist>` autocomplete from the Brndle icon set (same pattern as MegaMenu admin UI).
- BlockControls: density toggle, connector style select.

**Frontend UX:**
- `<ol class="brndle-timeline">` — semantic ordered list (timelines are ordered).
- Each `<li>` contains: dot/icon column on the left, content column on the right. Connector is a `:before` pseudo-element on the dot column at 1px (solid) or `border-style: dashed`.
- Per-item dot: when `iconStyle === 'numbered'`, renders the 1-based index in a small accent circle. When `iconStyle === 'lucide'`, renders the picked icon at 16px inside a 32px circle. When `iconStyle === 'dot'`, a 12px filled accent circle.
- Reveal animation: items fade-in + slide-in-from-left as they enter the viewport (Intersection Observer, 50px before viewport). Stagger of 60ms between siblings. Reduced motion: no animation, items are immediately at final state.
- Density `compact`: 1rem gap between items. `comfortable`: 2rem.
- Accessibility: dot column is `aria-hidden="true"` because it's purely decorative; the date and title carry the meaning. The `<ol>` role/structure is read correctly by screen readers.

**Edge cases:**
- Empty `items`: don't render.
- One item: connector hidden (`:only-child` rule).
- Mobile: stays vertical (timelines are always vertical in this design — no horizontal-scroll variant). Density forced to `compact` below `sm` regardless of attribute, to save vertical space.

**Tests:**
- Reveal animation runs once per item (no re-animations on scroll-back).
- Connector line aligns vertically through every dot (regression test against single-line vs multi-line item content).
- All 3 icon styles render correctly.
- Numbered style increments correctly when items are reordered.

### M4 — Tabs / Accordion (`brndle/tabs-accordion`)

**Reference:** Radix UI Tabs + Accordion primitives.

**What:** Combined block. Single attribute schema. `displayMode` toggle picks tab vs accordion rendering. Same data, different presentation.

**Why combined:** less menu noise, one editorial mental model. The two patterns share data shape (label + content panels).

**Attributes:**
- `title` (string, default `''`) — optional section title
- `items` (array of `{ label: string, content: string }`) — each panel
- `displayMode` (enum: `tabs`, `accordion`, default `tabs`)
- `tabsAlignment` (enum: `start`, `center`, `end`, default `start`) — only relevant for `tabs`
- `accordionMode` (enum: `single`, `multiple`, default `single`) — single closes others when one opens; multiple lets all be open
- `accordionDefault` (enum: `closed`, `first`, `all`, default `closed`) — initial state
- `uniqueId` (string)

**Editor UX:**
- Repeater pattern (one PanelBody per item).
- Each item's `content` is a `<TextareaControl>` (NOT inner blocks — keeps the data shape predictable for SSR). Limited HTML allowed via `wp_kses_post()` server-side (b, i, strong, em, a, br, code).
- BlockControls: displayMode toggle (tabs ↔ accordion).
- InspectorControls: alignment (tabs only), accordion mode, accordion default state.

**Frontend UX — Tabs mode:**
- WAI-ARIA Tabs pattern, faithful to the spec:
  - `<div role="tablist" aria-orientation="horizontal">` containing `<button role="tab" aria-selected aria-controls id>`
  - Panels: `<div role="tabpanel" aria-labelledby tabindex="0">` (tabindex 0 because panels have no inherent focusable content guaranteed)
  - Keyboard: ←/→ to move between tabs, Home/End to jump to first/last, Tab to enter the active panel.
  - Inactive tabs are `tabindex="-1"` so Tab cycles to the next focusable element rather than every tab.
- Visual: tabs use bottom-border style with accent color on active. Smooth indicator slide between tabs (200ms ease, neutralized under reduced motion to instant).
- Mobile (<640px): horizontal tab strip becomes scrollable with `scroll-snap-x` — never wraps. Active tab auto-scrolls into view.
- No tab content lazy load (data is present at render time; lazy-load is for blocks like media).

**Frontend UX — Accordion mode:**
- WAI-ARIA Disclosure pattern (NOT the broken "accordion" pattern — disclosure is the modern, accessible recommendation):
  - Each item: `<h3><button aria-expanded aria-controls></button></h3>` + `<div id role="region" aria-labelledby hidden>`
  - Keyboard: Enter/Space toggles. Tab cycles through buttons + open content normally.
- Animation: `max-height` transition from 0 to `scrollHeight` over 220ms (cubic-bezier ease). Reduced motion: instant snap (no transition).
- `accordionMode === 'single'`: opening one closes the others (radio-like).
- `accordionDefault === 'first'`: first item starts open.
- `accordionDefault === 'all'`: all open (only meaningful in multiple mode).

**No FAQPage schema.** Existing FAQ block owns that. Tabs/accordion is neutral.

**Edge cases:**
- Empty `items`: don't render.
- One item: accordion still works (single-disclosure). Tabs mode hides the tablist (just renders the panel), since a single tab is meaningless.
- Switch from tabs ↔ accordion in the editor: data preserved; CSS swaps; JS controller re-attaches.

**Tests (mandatory, axe-able):**
- Tab keyboard interaction matches WAI-ARIA spec exactly (←/→/Home/End cycle, Tab enters panel).
- Accordion Enter/Space toggle works; aria-expanded flips.
- Switch displayMode round-trip preserves data.
- Mobile tab strip scrolls horizontally without wrap.
- Reduced motion: no animations.

## Out of scope (do NOT build without specific request)

- Code block: dynamic language detection, multi-tab code (e.g. "JS / TS / Python" tabs in a code sample) — defer; can be composed using a tabs-accordion block wrapping multiple code blocks today.
- Pull quote: Twitter / X share button on quote (gimmicky, not editorial register).
- Timeline: drag-to-reorder in editor (overscope; up/down buttons cover 95% of the use case).
- Timeline: filter / search across timeline items (overscope; long timelines aren't part of the editorial use case).
- Tabs: lazy-load tab content from a URL (overscope; data is editor-authored).
- Tabs: nested tabs (Radix supports it; we don't need it).
- All four: per-block animations beyond what's specified (no "extra polish").
- Accordion: emit `FAQPage` JSON-LD even when used like FAQ — that's the FAQ block's job.

## Open decisions

1. **Code block highlighter strategy** → **decided**: client-side via highlightjs lazy-loaded from CDN. PHP-side and bundled-subset paths are filterable but not the default.
2. **Tabs/Accordion combined or separate** → **decided**: combined. One block, displayMode toggle.
3. **Timeline icon style** → **decided**: three modes baked in (`dot`, `numbered`, `lucide`). No "custom upload" — over-scope for editorial use.
4. **CodeMirror in editor canvas?** → **no**. Plain `TextareaControl` with monospace font. CodeMirror is too heavy for the editor canvas + brings JS bundle weight that v2.1 specifically avoids.

## Risk notes

- **highlightjs CDN dependency** — if the site is on a strict CSP that blocks `esm.sh`, the highlighter never loads. Filter `brndle/code_highlighter_strategy` lets sites switch to `'inline'` (bundled subset, ~30KB) or `'php'` (server-side). Document in the release notes.
- **WAI-ARIA Tabs pattern is famously easy to get wrong.** Test against axe-core programmatically; reference the official W3C Tabs Pattern. Don't ship until keyboard nav is verified.
- **`outset` pull quote breaks if the article column is narrower than 540px** — the CSS guards via `:where(.sidebar)` (the only narrow layout) but custom layouts may break. Document the constraint.
- **`viewScriptModule` vs `viewScript`** — the wp-block-development skill prefers `viewScriptModule` (ES module, deferred). Brndle's existing blocks all use `viewScript` via lazy-enqueue from the render callback. Stay consistent — the existing pattern works and changing it breaks the per-instance lazy-load logic in `BlockServiceProvider`. Stick with the lazy-enqueue pattern.
