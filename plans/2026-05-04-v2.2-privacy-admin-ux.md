# Plan: v2.2 — privacy + admin UX (premium UX)

**Date:** 2026-05-04
**Author:** Claude Code (with Varun)
**Effort:** ~10–12h across 4 milestones

This plan covers v2.2 of the post-1.9.2 audit roadmap. Four pieces — three sit on privacy (cookie consent, click-to-load embed) + admin UX (pattern showcase, Site Health) — built to the same premium UX bar as v2.0 + v2.1 already shipped.

## Goals

1. **Privacy parity with paid themes.** Native cookie consent banner that doesn't compete with installed consent plugins. Click-to-load embed wrapper that defers third-party iframes (YouTube / Vimeo / Loom / generic) until user opt-in.
2. **Admin UX parity with the page-builder ecosystem.** A pattern-showcase page editors use to discover blocks; Site-Health tests that surface Brndle-specific config drift before it bites.
3. **Zero new heavy dependencies.** Total JS budget across all four ≤ 5KB gzipped (cookie consent + click-to-load combined). No third-party libraries. Pattern showcase is admin-only — frontend stays untouched.
4. **All four respect `prefers-reduced-motion`, full WAI-ARIA on interactive surfaces, RTL-safe.**

## Constraints (non-negotiable)

From `wp-content/themes/brndle/CLAUDE.md`:
- Tailwind v4 utilities + Brndle theme tokens. No `text-[var(--color-*)]`.
- Editor JSX → `@wordpress/icons` (auto tree-shaken). Frontend Blade → `<x-icon>` (Lucide).
- No emoji as UI affordance.
- No jQuery anywhere. View scripts vanilla, idempotent, lazy-loaded.

From the `wp-block-development` + `wp-plugin-development` skills:
- Per-instance scope id for any feature whose CSS/JS isn't fully utility-driven (cookie banner, click-to-load wrapper).
- `prefers-reduced-motion` neutralizes every transition.
- Theme isolation — every CSS rule scoped to its feature class.
- BEM-ish class naming: `.brndle-{feature}__{element}--{modifier}`.

From memory:
- **Defer to consent plugins on every Brndle site** (`feedback_brndle_seo_plugin_defer.md` rule). When CookieYes / Cookie Notice / Complianz / GDPR Cookie Consent is active, suppress the theme's banner entirely and log an admin notice. Same shape as the Yoast / Rank Math defer rule from v2.0.
- **No SEO schema competition** — no Article / BreadcrumbList / Person JSON-LD. v2.0's defer rule still applies.
- **No versions in plans** — describe milestones (M1/M2/M3/M4), not version numbers. Version gets assigned at release time.
- **Long-term solutions over per-instance patches** — build a shared `Privacy/ConsentStore` for the cookie + embed gate to read from, rather than two parallel consent pipelines.

## Aesthetic direction

Two distinct registers in this bundle, used in different surfaces:

- **Cookie consent + click-to-load embed (frontend, user-facing):** restrained, minimum viable. The banner shouldn't feel like a marketing-register block. Hairline border, single-line message at desktop, two clean buttons (Accept / Decline). Click-to-load placeholder shows the YouTube poster image with a centered play button + a one-line "Load video — sets a YouTube cookie" affordance. Boring is correct here.
- **Pattern showcase + Site Health (admin, editor-facing):** WordPress-admin-native. Use `@wordpress/components` (PanelBody, Card, Notice, Button). Lean on the existing Brndle settings panel chrome. The pattern showcase scrollable page should feel like Storybook — block name + tag chips + the rendered block + a "Copy block markup" button.

Reference designs (each milestone picks 2 as North Star):

- **Cookie banner**: Stripe's cookie banner, GitHub's cookie banner, Linear's cookie banner. All small, footer-pinned, fast to dismiss.
- **Click-to-load**: Plausible Analytics docs (YouTube embeds), DuckDuckGo's tracker shield UI, Vercel docs.
- **Pattern showcase**: Storybook (canonical), Radix UI primitives docs, Polaris (Shopify) component browser.
- **Site Health**: WP core Site Health UI is the contract — extend it with our tests, don't reinvent.

## Cross-cutting infrastructure

Built once, used by M1 + M4 (privacy register):

### Shared: `Brndle\Privacy\ConsentStore`
`app/Privacy/ConsentStore.php`. Static-method facade over the consent state.

- `ConsentStore::isPluginActive(): bool` — detect CookieYes (`function_exists('CKY_LITE_VERSION')` or `class_exists('Cky_Lite\Public\Cky_Lite_Public')`), Cookie Notice (`class_exists('Cookie_Notice')`), Complianz (`function_exists('cmplz_user_consent')` or `defined('cmplz_version')`), GDPR Cookie Consent (`class_exists('GDPR_Cookie_Consent')`). Cached for the request.
- `ConsentStore::userConsented(string $category = 'analytics'): bool` — read `localStorage` flag (server-side via cookie shadow) OR plugin's API when active. Categories: `functional` (always true), `analytics`, `marketing`. Filter `brndle/consent_categories`.
- `ConsentStore::userDeclinedDNT(): bool` — `navigator.doNotTrack === '1'` echoed to a cookie by the consent JS so PHP can read it. Auto-decline path.
- `ConsentStore::adminNotice(): void` — emits the "consent plugin detected, theme banner suppressed" admin notice on the consent plugin defer path. Same shape as v2.0's `SeoPluginNotice`.

### Shared: `resources/js/privacy/consent-store.js` (vanilla, ~600 bytes gzipped)
Mirror of the PHP store. Reads + writes `brndle_consent` cookie + `brndle-consent` localStorage. Exports:
- `getConsent(category)` — returns boolean
- `setConsent(category, value)` — writes + dispatches a `brndle:consent-changed` event
- `dntDeclined()` — reads `navigator.doNotTrack`

The cookie banner JS (M1) and the click-to-load embed JS (M4) both import from this. Single source of truth for consent state on the page.

### Shared: shared admin scaffolding for M2 + M3
- M2 (pattern showcase) and M3 (Site Health) both add admin-only assets. Both register through the existing `admin/src/` React entry but mount on different admin pages.
- Pattern showcase is mounted via a new top-level admin item under `Brndle → Pattern showcase`.
- Site Health tests hook into core's `site_status_tests` filter — pure PHP, no React. Lives in `app/SiteHealth/`.

## Per-milestone specs

### M1 — Cookie consent

**Reference:** Stripe + GitHub footer banners.

**What:** Native cookie banner. Tiny. Respects DNT. Defers entirely when a known consent plugin is active.

**Architecture:**
- `app/Privacy/CookieConsent.php` — the controller (boots the JS + renders the banner partial when no consent plugin is active and consent isn't already given).
- `app/Privacy/ConsentStore.php` — shared store (see Cross-cutting).
- `resources/views/partials/privacy/consent-banner.blade.php` — the banner markup.
- `resources/js/privacy/consent-banner.js` — the controller (~1KB gzipped). Mounted from `app.blade.php` only when the banner partial rendered.

**Banner UX (default — minimal mode):**
- Position: fixed, bottom of viewport, centered max-w-3xl.
- Single-line copy at desktop: "We use cookies to improve this site. You can decline non-essential cookies."
- Two buttons: `Accept` (primary, accent bg + white text) and `Decline` (ghost, text-secondary).
- Optional third button "Manage" (only when filter `brndle/consent_categories` returns >1 category) → opens an inline sub-panel with checkboxes for each category.
- Hairline top border separating from page; `bg-surface-primary` with subtle box-shadow upward.
- Focus-trap: Tab cycles between the banner buttons until Accept / Decline closes it.
- Slide-up entrance (200ms ease, 300ms delay) so it doesn't compete with first-paint LCP. Reduced-motion: instant, no slide.
- Mobile (<640px): two-line layout — copy on top, buttons below stacked horizontally.

**Categories filter:**
```php
add_filter('brndle/consent_categories', function () {
    return [
        'functional' => ['label' => 'Functional', 'required' => true],
        'analytics' => ['label' => 'Analytics', 'required' => false],
        'marketing' => ['label' => 'Marketing', 'required' => false],
    ];
});
```
Default: just `functional` (required) + `analytics` (optional). Sites that need GDPR-strict granularity can add `marketing`.

**DNT path:**
- On first paint (before banner shows), JS reads `navigator.doNotTrack`. If `'1'`:
  - Write `brndle_consent` cookie with all non-required categories declined
  - Skip rendering the banner entirely
  - Dispatch `brndle:consent-changed` so the click-to-load embed (M4) reads the declined state

**Plugin-defer path:**
- `CookieConsent::boot()` checks `ConsentStore::isPluginActive()`. If true:
  - Skip enqueuing the consent JS
  - Skip rendering the banner partial
  - Show admin notice via `Brndle\Privacy\ConsentPluginNotice` (one-time dismissible) telling the user we're deferring to their installed plugin
- The click-to-load embed (M4) still works — it just reads consent from the plugin's API instead of the theme's localStorage.

**Filters:**
- `brndle/consent_categories` — categories array (above)
- `brndle/consent_banner_copy` — override default copy (string or array `['title' => ..., 'body' => ..., 'accept' => ..., 'decline' => ...]`)
- `brndle/consent_banner_position` — `'bottom'` (default) | `'top'`
- `brndle/consent_plugin_detected` — extension hook for detecting custom consent plugins not in the bundled detection list

**Edge cases:**
- User accepts → `brndle_consent` cookie set with `analytics: true`, banner slides down + removes from DOM, `brndle:consent-changed` event fires.
- User declines → cookie set with `analytics: false`, banner removed, `brndle:consent-changed` event fires with declined categories.
- Cookie already set on page load (returning visitor) → banner never renders.
- DNT enabled → banner never renders, all non-required declined automatically.
- Consent plugin active → banner never renders, admin notice visible until dismissed.
- JS disabled → banner never renders (it's CSS-hidden by default + JS adds `is-visible` class). Site stays GDPR-compliant because no analytics scripts fire — they're gated server-side via `ConsentStore::userConsented()`.

**Tests:**
- Banner appears bottom-fixed on first visit
- Accept → cookie set + banner removed + event dispatched
- Decline → cookie set with `analytics: false` + banner removed
- DNT enabled → no banner rendered
- CookieYes / Cookie Notice / Complianz / GDPR-CC active → no banner + admin notice visible
- Returning visitor with cookie → no banner
- Mobile 390px: copy stacks above buttons; buttons fit
- Reduced-motion: instant slide
- Banner click-outside doesn't dismiss (intentional — must accept or decline)

**Effort:** ~2.5h (slightly over the 2h roadmap estimate due to plugin-defer + DNT + categories filter)

### M2 — Pattern showcase admin page

**Reference:** Storybook + Polaris component browser.

**What:** A scrollable admin page showing every Brndle block + every color scheme + every font pair, rendered live. Editors / clients use it to discover what's available; agencies use it for client demos.

**Architecture:**
- New admin page under `Brndle → Pattern showcase` (top-level menu sibling).
- New React entry `admin/src/pattern-showcase.js` + new webpack entry.
- Mounts on the page via the existing Brndle React infrastructure.
- Reads block data via `wp.blocks.getBlockTypes()`. Renders each block via `ServerSideRender` with the example attributes from each block's `block.json`.
- Color schemes + font pairs sourced from `Brndle\Settings\ColorPalette::all()` + `Brndle\Settings\FontPairs::all()`.

**Page UX:**
- Top bar: "Brndle Pattern Showcase" + filter pills (All / Marketing / Editorial) + a search input that filters by block name/keyword + a "Copy block markup" toggle (when on, every block has a clipboard-copy affordance).
- Body: vertical stack of every block, one per section. Each section:
  - Block name (h2) + category chip + keyword chips
  - Description from `block.json`
  - Live render (using `ServerSideRender` with `example.attributes` from the block.json)
  - "Copy block markup" button that copies the inserter-equivalent JSON
  - "Open in editor" link that creates a draft post pre-filled with this block
- Sticky sidebar (right): mini-ToC with anchors to each block. Click → smooth-scroll. Active item highlights as user scrolls.
- Footer of each block section: small "Used in: 14 single-post layouts / Landing template / etc." metadata pulled from a static map.

**Color/font preview slot:**
- Top of page (above the block list): "Theme tokens" panel showing all 12 color schemes as swatches + all 8 font pairs as type specimens.
- Each token clickable: clicking a color scheme temporarily applies it to the showcase preview area (data-attribute swap on a wrapper, doesn't affect saved settings). "Reset to current" button to revert.

**Filters:**
- `brndle/pattern_showcase_blocks` — array of block names to include (default: all 18)
- `brndle/pattern_showcase_extra_metadata` — extra metadata per block ("used in" data, badges, etc.)

**Edge cases:**
- Block with required attributes but no example → renders an empty-state placeholder with the message "No example data; add `example` to `block.json`".
- Server-side rendering fails for a block (missing render callback / template) → graceful empty state with the error message.
- Block list >50 items → virtualize the render via `IntersectionObserver` (only render blocks within ~2 viewport heights of view) so the page stays responsive.

**Accessibility:**
- ToC sidebar uses `<nav aria-label="Pattern showcase navigation">`
- Filter pills are `<button role="radio">` inside `<div role="radiogroup">`
- Search input has `aria-controls` pointing at the block list
- Each block section has `id` for ToC linking + `tabindex="-1"` so smooth-scroll-to-anchor lands focus

**Effort:** ~3.5h (over the 3h roadmap estimate due to React virtualization + color/font preview slot)

### M3 — Brndle-specific Site Health checks

**Reference:** WP core Site Health UI.

**What:** Hook `site_status_tests` filter and add Brndle-specific async + direct tests so admins surface config drift before users hit it.

**Architecture:**
- `app/SiteHealth/Tests.php` — pure PHP. Boots from `ThemeServiceProvider`.
- Each test is a method returning the standard WP core Site Health response shape: `['label' => ..., 'status' => ..., 'description' => ..., 'actions' => ..., 'test' => ...]`.
- `status` enum: `'good'` (passing) | `'recommended'` (low-priority issue) | `'critical'` (high-priority issue).

**Tests shipped (default):**

| ID | Label | Severity if failing | Description on fail |
|---|---|---|---|
| `brndle_critical_css` | Critical CSS file present | recommended | "Brndle's critical CSS feature is enabled, but `resources/css/critical.css` is missing or unreadable. The page will fall back to render-blocking app.css until the file is restored. Disable the feature in Brndle → Performance, or rebuild the theme with `npm run build`." |
| `brndle_settings_consistency` | Brndle settings + admin UI in sync | critical | "The settings consistency CI is failing — at least one Brndle setting in `Defaults::all()` has no admin UI control or vice versa. Run `node bin/check-settings-consistency.mjs` and check the output." |
| `brndle_primary_nav_set` | Primary navigation menu assigned | recommended | "Brndle expects a `primary_navigation` menu to be set. The header currently shows no nav. Assign a menu under Appearance → Menus → Manage Locations." |
| `brndle_image_sizes_registered` | Brndle image sizes registered | recommended | "Brndle registers `brndle-hero` (1600×900) and `brndle-card` (800×600) image sizes for the hero and feature blocks. One or more sizes are missing — likely a child theme override removed `add_image_size`. Regenerate via the Regenerate Thumbnails plugin." |
| `brndle_seo_plugin_active` | SEO plugin (Yoast or Rank Math) active | recommended | "Brndle defers Article / Person / Breadcrumb structured data to an SEO plugin. Without one, posts publish without structured data. Install Yoast SEO or Rank Math." |
| `brndle_view_cache_writable` | Acorn view cache writable | critical | "The Acorn Blade view cache directory at `wp-content/cache/acorn/framework/views/` isn't writable by the web server. Pages will fail to render until permissions are fixed: `chmod -R 755 wp-content/cache/acorn/`." |

**Filter:**
- `brndle/site_health_tests` — extend the test list. Custom tests can return the same response shape.

**Effort:** ~2h (matches roadmap estimate)

**Edge cases:**
- Tests run on Site Health page load — must be cheap. Each test ≤ 50ms. Heavy probes (e.g., re-running the settings consistency CI) are run async via the WP cron + cached for 12h.
- A test that crashes with a fatal error breaks the entire Site Health page. Wrap each test in `try/catch` and return a `['status' => 'critical']` response with the exception message if it throws.

### M4 — Click-to-load embed wrapper

**Reference:** Plausible Analytics docs YouTube embeds.

**What:** A Blade component `<x-embed>` (and matching block wrapper later) that takes a YouTube / Vimeo / Loom / generic iframe URL. Renders a poster image + play button on first paint. Iframe injected only when the user clicks (or when consent is already given via M1).

**Architecture:**
- `resources/views/components/embed.blade.php` — the Blade component.
- `app/Privacy/EmbedRegistry.php` — static map of provider → poster URL extractor + iframe URL builder.
- `resources/js/privacy/click-to-load.js` — the click handler (~600 bytes gzipped). Reads `ConsentStore` (M1 shared store) and:
  - If consent already given for `marketing` (or whichever category the embed is tagged with), inject iframe immediately on page load
  - Otherwise, render poster + play button. On click, inject the iframe AND set consent for the embed's category.

**Component API:**
```blade
<x-embed
  url="https://www.youtube.com/watch?v=dQw4w9WgXcQ"
  poster="https://example.com/poster.jpg"
  caption="Watch the 90-second demo."
  category="marketing"
  aspect-ratio="16/9"
/>
```

**Provider auto-detection:**
- YouTube: `https://www.youtube.com/watch?v=ID` or `https://youtu.be/ID` → `https://www.youtube.com/embed/ID` + auto-fetch poster from `https://i.ytimg.com/vi/ID/maxresdefault.jpg`
- Vimeo: `https://vimeo.com/ID` → `https://player.vimeo.com/video/ID` + auto-fetch poster via Vimeo oEmbed (cached 24h)
- Loom: `https://www.loom.com/share/ID` → `https://www.loom.com/embed/ID` + manual poster (Loom's oEmbed is paywalled)
- Generic iframe: caller provides full embed URL + poster

**Frontend UX:**
- Wrapper: `<div class="brndle-embed" data-brndle-embed style="aspect-ratio: 16/9">`
- Default state: `<img class="brndle-embed__poster" src="{poster}" alt="">` + `<button class="brndle-embed__play" aria-label="Load video — sets a {provider} cookie"><x-icon name="play-circle" /></button>` + small caption strip at bottom: "Click to load — sets a {provider} cookie"
- After click: poster + button removed, iframe injected with `loading="eager" allowfullscreen`. Caption replaced with a simple "{caption}" text (no consent disclaimer, since user just consented).
- If consent already given on page load: skip the poster step entirely, render iframe directly.
- Reduced-motion: no fade between states; instant swap.

**Caption / accessibility:**
- The full unloaded state is fully keyboard-accessible: Tab focuses the play button, Enter/Space activates.
- The "sets a {provider} cookie" disclaimer is part of the `aria-label` so screen readers announce the privacy implication before activation.
- After load, `<iframe title="...">` carries the caption so screen readers know what loaded.

**Lucide icons added:** `play-circle` (or `play-circle-2`) — confirm with the curated list.

**Filters:**
- `brndle/embed_providers` — extend the provider registry (caller adds new domain → URL builder pairs)
- `brndle/embed_default_category` — default consent category for unspecified embeds (default: `marketing`)

**Effort:** ~2h (slightly over roadmap's 1.5h to cover oEmbed poster fetching for Vimeo + the iframe injection lifecycle)

**Edge cases:**
- YouTube ID extraction fails (URL format we don't recognize) → embed renders as a plain `<a href>` link with the URL — never fail silently.
- Poster URL returns 404 → falls back to a CSS gradient + play button. Don't block rendering on the poster.
- Vimeo oEmbed timeout (3s) → falls back to a CSS gradient.
- User has consent for `marketing` → embed loads on first paint without poster step. Same UX as a regular embed.
- User declines `marketing` later → already-loaded iframes stay loaded (don't unload mid-page; just don't load NEW ones in the same session).

**Block integration (out of v2.2 scope, noted for future):**
The existing `brndle/video-embed` block currently embeds iframes directly. v2.2 introduces the `<x-embed>` component but does NOT migrate the existing block. A future minor will add a `useClickToLoad` toggle to that block; v2.2 ships the underlying primitive only.

## Out of scope (do NOT build without specific request)

- **Cookie consent**: server-side IP geolocation (only show banner in EU). Out of scope; let the site owner add this via filter if they want.
- **Cookie consent**: full GPC (Global Privacy Control) signal handling. DNT covers 90% of the use case; GPC adds complexity for small marginal gain.
- **Pattern showcase**: drag-and-drop reordering of blocks in the showcase. Editor only — too much UI for a discovery surface.
- **Pattern showcase**: per-client custom block libraries. Defer until 5+ agencies ask for it.
- **Site Health**: write a Brndle-specific dashboard. The existing WP core Site Health page is the right surface; don't reinvent.
- **Click-to-load**: lazy-loading non-iframe content (Twitter embeds, Instagram embeds). YouTube + Vimeo + Loom + generic iframe is 95% of the actual use case; the rest can be added incrementally.
- **Click-to-load**: server-side static thumbnail generation (Vimeo poster fetch). Out of scope; cache the oEmbed response and accept the 3s timeout fallback.

## Open decisions (resolve at start of M*)

**M1 — Cookie consent:**
1. Banner position default: top vs bottom? → **bottom** (Stripe / GitHub / Linear all use bottom; matches modern norm).
2. Default categories: `[functional, analytics]` or `[functional, analytics, marketing]`? → **2-category default** (`functional` + `analytics`). Sites that need marketing-tier granularity add via filter.
3. Banner copy customization: how granular? → 4 strings (`title`, `body`, `accept`, `decline`) overridable via single filter `brndle/consent_banner_copy`.
4. Should declining consent immediately clear any already-set analytics cookies? → **No.** That's the analytics plugin's job, not ours.

**M2 — Pattern showcase:**
1. Admin menu placement: top-level "Brndle → Pattern showcase" or sub-page under "Brndle Settings"? → **top-level**. Discovery surface; deserves its own slot.
2. Color/font preview: live data-attribute swap (no save) or open in customizer? → **live swap**. Faster iteration; no redirect.
3. Block search: client-side filter or server-side query? → **client-side**. 18 blocks is small; instant feedback wins.

**M3 — Site Health:**
1. Tests cached vs live? → **6 default tests are all live (cheap)**. The settings consistency CI is async-cron-cached for 12h since it shells out to Node.
2. How to surface "this is failing on a child site I don't manage" — opt-out filter for tests? → **`brndle/site_health_tests`** filter lets sites remove individual tests by ID.

**M4 — Click-to-load embed:**
1. Poster fetching strategy: build-time, server-time, or client-time? → **server-time with 24h transient cache**. YouTube has a deterministic poster URL pattern (no fetch needed); Vimeo + Loom need oEmbed (server-time).
2. Default category for embeds: `analytics` or `marketing`? → **`marketing`**. Most strict GDPR interpretation; ad-supported networks like YouTube fall into this category by default.

## Risk notes

- **Cookie consent regulatory drift.** GDPR + ePrivacy are unstable surfaces. Brndle's banner is a baseline — sites in heavily regulated jurisdictions (Germany, France) should still install a dedicated consent plugin. The plugin-defer detection logic must work reliably; document that as a hard requirement.
- **Pattern showcase performance with 100+ blocks.** Today there are 18 blocks. If the catalog grows to 50+ via child themes / addons, server-side rendering all of them on a single page becomes slow. Build the virtualization in M2 from the start, even though 18 fits on one page comfortably.
- **Site Health false positives on cron-cached tests.** Async tests (settings consistency CI) cache failures for 12h. If a fix lands within those 12h, the test stays red. Add a "Re-run check" button on the test detail page that bypasses the cache.
- **Click-to-load + browser back-button.** Some browsers cache the iframe-loaded state when navigating back; users see the iframe even though they didn't click. This is correct behavior (they consented in the same session) but may surprise. Consider using `pageshow` event with `event.persisted` to re-evaluate consent on bfcache restore.
- **Vimeo oEmbed paywall.** Vimeo's free tier oEmbed throttles aggressively (30 req/min). Cache hard with a 24h transient. Sites with many Vimeo embeds should provide explicit `poster` attrs.
